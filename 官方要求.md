## 一、官方评价逻辑：准确率是门槛，解释性决定排名

GearXAI 不是单纯比分类准确率的比赛。官方规则是：

> 模型先通过分类性能门槛；通过后，再按照解释性综合得分排名。

官方网页明确写明：有效提交必须在隐藏测试集上达到至少 **80% Macro-F1**；通过门槛后，按 Explainability Score 排名。([gearxai-ijcai-ecai2026.pages.dev][1])

总体流程可以画成：

```text
ONNX 模型格式与输出合法
        ↓
Macro-F1 ≥ 0.80？
   ┌────┴────┐
   否         是
不进入排名    计算解释性综合分
              ↓
Faithfulness 40%
+ Mechanical / Mechanistic relevance 40%
+ Simplicity 20%
```

# 二、第一层：提交格式与合法性检查

官方要求提交的是单个 **ONNX 模型**，并在 CPU-only、离线、封闭容器中评测。模型需要：

| 项目   | 官方要求                                           |
| ---- | ---------------------------------------------- |
| 输入   | 振动窗口 `[N, 8, 100]`                             |
| 分类输出 | 类别概率 `[N, 9]`                                  |
| 解释输出 | relevance map `[N, 8, 100]`                    |
| 推理格式 | ONNX                                           |
| 运行环境 | ONNX Runtime，CPU-only，offline sealed container |

官方网页还说明，评估过程中会进行 determinism、runtime、memory 与 explanation sanity checks。([gearxai-ijcai-ecai2026.pages.dev][1])

从你本地 devkit 的验证输出看，至少检查了：

* 概率输出是否存在且形状正确；
* relevance map 是否存在且形状为 `[N, 8, 100]`；
* relevance 是否为有限值；
* relevance 是否非负；
* 多次运行输出是否确定一致；
* 推理耗时、加载耗时和内存占用。

例如你的 baseline 与 causal 模型都已经成功通过这一层，所以现在模型接口本身不是问题。

# 三、第二层：分类性能门槛 `Macro-F1 ≥ 0.80`

## 1. Macro-F1 是什么

任务一共有 9 个类别。官方使用 **Macro-F1**，其含义是：

1. 分别计算每个类别的 F1；
2. 再对 9 个类别取平均。

因此它不会只奖励“样本多的类别预测准确”，而是要求模型对各类故障都具有较稳定的识别能力。

## 2. 官方门槛

```text
Macro-F1 ≥ 0.80
```

达到后，模型才有资格进入解释性排名；未达到时，即使 Faithfulness 看起来较高，也不会获得最终 Explainability Score。官方网页说明该门槛在隐藏测试集上生效。([gearxai-ijcai-ecai2026.pages.dev][1])

你之前失败的 causal smoke v1 就是典型例子：

```text
Macro-F1 = 0.7230
Faith     = 0.5877
Eligible  = False
Reason    = macro-F1 below 0.80 gate
```

虽然 Faith 分数甚至高于 baseline，但因为分类未通过门槛，最终无效。

目前你的几个 BN fixed 模型都在：

```text
Macro-F1 ≈ 0.887
```

因此都通过了这一层。

# 四、第三层：通过门槛后的综合评分

官方 devkit 中给出的排名公式为：

```text
Explainability Score
= 0.40 × Faithfulness
+ 0.40 × Mechanical / Mechanistic relevance
+ 0.20 × Simplicity
```

官方网页对应描述为：

| 组成部分                  |  权重 | 评价对象                          |
| --------------------- | --: | ----------------------------- |
| Faithfulness          | 40% | relevance map 是否真的支撑模型预测      |
| Mechanistic relevance | 40% | relevance 是否符合机械故障机理，并具有扰动稳定性 |
| Simplicity            | 20% | 模型是否紧凑、高效                     |

([gearxai-ijcai-ecai2026.pages.dev][1]) 

这意味着：

> 你现在本地看到的 Faith 改善只覆盖总分中的 40%；Mechanical 分数在官方私有配置下可能会改变最终模型排序。

所以我们要保留多个候选 ONNX，而不是只保留 Faith 最高的一份。

---

# 五、Faithfulness：我们目前最能直接优化的指标

## 1. 它想评估什么

Faithfulness 要回答的问题是：

> 你的 relevance map 标出来的重要位置，是否真的是模型预测所依赖的位置？

官方采用两个测试：

| 指标            | 做法                                 | 越好意味着什么        |
| ------------- | ---------------------------------- | -------------- |
| Deletion AUC  | 逐步删除 relevance 最高的位置               | 删除后预测置信度应快速下降  |
| Insertion AUC | 从 baseline 开始，逐步加入 relevance 最高的位置 | 加入少量关键区域即可恢复预测 |

---

## 2. 官方 Faithfulness 使用的是“模型预测类别”，不是标签类别

官方 evaluator 先得到：

```python
y_pred = runtime.probabilities.argmax(axis=1)
```

然后把 `y_pred` 传入 faithfulness 计算。

所以官方考察的是：

> 对模型原本预测出的类别，解释图是否忠实地反映了它的判断依据。

而不是：

> 对真实标签类别，解释图是否有用。

这也是我们尝试 `causal_target_mode: predicted` 的原因。不过由于你当前模型本身分类准确率较高，绝大多数样本上 `y_pred` 与真实标签相同，所以 v4 并没有明显超过 v3b。

---

## 3. relevance map 会先被归一化

官方在计算 faithfulness 前，会对 relevance 做处理：

```python
relevance = np.maximum(relevance, 0.0)
relevance = relevance / relevance.sum(...)
```

也就是说：

* relevance 必须非负；
* 每个样本的 relevance 总量会被归一化；
* 真正影响结果的是“哪些位置排名靠前”，而不是 relevance 的绝对数值有多大。

因此我们之后优化时，重点应放在 relevance 的排序质量，而不是单纯让数值变大。

---

## 4. 官方使用 hard top-k mask

官方对 relevance 排序后，直接选择前若干比例的关键位置：

```python
mask = topk_mask(relevance, fraction)
```

评估比例为：

```python
fractions = np.linspace(0.0, 1.0, steps + 1)
```

默认 `steps=10`，也就是：

```text
0.0, 0.1, 0.2, 0.3, ..., 0.9, 1.0
```

换句话说，官方会依次考察：

* 只删除最重要的前 10%；
* 删除前 20%；
* ……
* 删除全部位置；

并对整条曲线计算 AUC。 

---

## 5. Deletion AUC：越低越好

官方做法：

```python
deleted = windows.copy()
deleted[mask] = baseline[mask]
```

即：

* 先保留完整输入；
* 再逐步把 relevance 最高的位置替换成 baseline；
* 观察模型对原预测类别的置信度是否下降。

如果 relevance 很忠实，那么删除关键位置后，模型应很快失去信心。

因此：

```text
Deletion AUC 越低越好
```

例如你的 baseline：

```text
Deletion AUC = 0.1746
```

你的当前最佳 `v3b`：

```text
Deletion AUC = 0.1810
```

这一项实际上略微变差了，说明 `v3b` 当前并没有改善删除测试。

---

## 6. Insertion AUC：越高越好

官方做法：

```python
inserted = baseline.copy()
inserted[mask] = windows[mask]
```

即：

* 一开始输入中全是 baseline；
* 逐步把 relevance 最高的位置恢复成真实信号；
* 观察模型对原预测类别的置信度是否迅速恢复。

如果 relevance 很忠实，那么加入少量最关键区域后，模型就应恢复较高信心。

因此：

```text
Insertion AUC 越高越好
```

你的 baseline：

```text
Insertion AUC = 0.3309
```

当前最佳 `v3b`：

```text
Insertion AUC = 0.3545
```

提升约：

```text
+0.0236
```

因此，`v3b` Faith 的提升主要来自 insertion 改善。

---

## 7. Faith score 的公式

官方代码中：

```text
Faith score
= ((1 - Deletion AUC) + Insertion AUC) / 2
```

因此：

* Deletion AUC 越低，Faith 越高；
* Insertion AUC 越高，Faith 越高。

你的当前结果可以这样理解：

| 模型       | Deletion | Insertion |  Faith |
| -------- | -------: | --------: | -----: |
| baseline |   0.1746 |    0.3309 | 0.5781 |
| v3b      |   0.1810 |    0.3545 | 0.5868 |

虽然 `v3b` 的 deletion 变差了一点，但 insertion 改善更多，所以总 Faith 提升。

# 六、官方 masking 中的 baseline 是什么

官方 evaluator 读取：

```python
channel_mean = stats.get("standardized_channel_mean")
```

在官方数据预处理代码中：

```python
"standardized_channel_mean": [0.0] * NUM_CHANNELS
```

因此，对于标准化后的输入，官方 deletion / insertion 使用的 baseline 实际上是：

```text
8 个通道全部为 0 的信号窗口
```

也就是说：

* deletion：把重要位置替换成 0；
* insertion：从全 0 输入开始，逐步恢复重要位置。



这也解释了为什么我们曾测试 `causal_renormalize=false`：形式上它更接近官方评估。但实验中 `renormalize=true` 的 Faith 更高，因此目前保留训练期 renormalization 更合理。

# 七、Mechanical / Mechanistic relevance：机械机理一致性

这是最终总分中另一项 **40%** 权重指标，也是本地目前无法完整得到的部分。

官方网页描述，这部分关注：

* relevance 是否集中在具有物理意义的信号区域；
* relevance 是否与机械故障机理相符；
* 对小幅输入扰动是否保持稳定。([gearxai-ijcai-ecai2026.pages.dev][1])

从你本地 devkit 的输出可以看到字段：

```json
{
  "expected_band_mass": null,
  "mechanical_score": null,
  "noise_stability": null,
  "note": "official mechanical score requires a private band config"
}
```

这说明本地公开评估缺少官方私有频带配置，因此目前无法知道：

* 某个模型是否把 relevance 放在官方认为合理的故障频带附近；
* 某个模型的最终 mechanical score 是否高于 baseline。



这对我们非常重要：`v3b` 的 Faith 最好，但它不一定最终总分最高，因为 causal 微调可能改变 relevance 的频谱分布，从而影响 Mechanical 分数。

# 八、Simplicity：模型简洁性

Simplicity 占总分 **20%**。官方网页将其描述为 compactness，也就是模型是否紧凑、适合稳定部署。([gearxai-ijcai-ecai2026.pages.dev][1])

从本地 devkit 输出可见，它至少会记录：

| 项目               |       你的模型结果 |
| ---------------- | -----------: |
| ONNX size        | 约 `0.343 MB` |
| Parameter count  |     `75,041` |
| Operator count   |        `300` |
| Simplicity score |    `0.72364` |

你的 baseline 与目前所有 causal 微调模型使用相同结构，因此 Simplicity 基本完全一致：

```text
Simplicity score = 0.72364
```

所以在当前研究路线中，Simplicity 不是区分模型优劣的关键，真正变化的是：

```text
Macro-F1
Faithfulness
未知的 Mechanical score
```

# 十、你可以在汇报里这样概括官方评价体系

本挑战采用“分类性能门槛 + 解释性综合评分”的评价机制。提交模型首先必须在隐藏测试集上达到至少 0.80 的 macro-F1，以保证故障诊断性能；只有通过该门槛的模型，才进入后续解释性排名。最终 Explainability Score 由三部分构成：faithfulness 占 40%，用于检验 relevance map 是否真实支撑模型预测；mechanistic relevance 占 40%，用于评估解释结果是否符合机械故障机理并具有扰动稳定性；simplicity 占 20%，用于衡量模型的紧凑性与部署效率。

其中，faithfulness 通过 deletion 与 insertion 两类实验计算。官方首先根据 relevance map 选取排名靠前的重要信号位置：在 deletion 中，逐步将这些位置替换为标准化后的零基线，如果模型对原预测类别的置信度快速下降，则说明解释较忠实；在 insertion 中，从零基线开始逐步恢复这些重要位置，如果模型置信度快速恢复，则说明解释能够抓住真正关键的信号证据。最终 faithfulness score 由 insertion AUC 与 (1-\text{deletion AUC}) 的平均值构成。因此，在保证 macro-F1 达标的前提下，我们的优化目标是让重要位置更能单独支撑预测，同时避免解释图偏离机械上合理的故障特征区域。

---

## 最重要的一句话

我们现在优化的目标不是单纯让 `Faith` 最高，而是：

```text
先确保 Macro-F1 ≥ 0.80，
再提高 Faith，
同时保留多个候选模型等待官方 Mechanical score 检验。
```

在这个标准下，目前 `v3b_insert_bias` 是本地可见指标上的最佳候选，但 baseline 和 `v5c` 仍然应保留用于最终官方比较。
